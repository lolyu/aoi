## 2. How SONiC Orchagent Creates These Objects

`TunnelDecapOrch` (in `sonic-swss/orchagent/tunneldecaporch.cpp`) responds to
`TUNNEL_DECAP_TABLE` and `TUNNEL_DECAP_TERM_TABLE` from APP_DB.

### Tunnel Object (SAI_OBJECT_TYPE_TUNNEL)

Created once per tunnel name. Key attributes:
- `SAI_TUNNEL_ATTR_TYPE` = `SAI_TUNNEL_TYPE_IPINIP` (always for decap tunnels)
- `SAI_TUNNEL_ATTR_OVERLAY_INTERFACE` = loopback RIF (created per-tunnel)
- `SAI_TUNNEL_ATTR_UNDERLAY_INTERFACE` = global underlay RIF
- `SAI_TUNNEL_ATTR_ENCAP_SRC_IP` = loopback IP (optional, set when src_ip configured)
- `SAI_TUNNEL_ATTR_DECAP_ECN_MODE` = copy_from_outer / standard
- `SAI_TUNNEL_ATTR_DECAP_TTL_MODE` = uniform / pipe
- `SAI_TUNNEL_ATTR_DECAP_DSCP_MODE` = uniform / pipe

The tunnel object itself is **metadata only** — it defines the encap/decap behavior
but does NOT trigger any dataplane programming by itself.

### Tunnel Term Table Entry (SAI_OBJECT_TYPE_TUNNEL_TERM_TABLE_ENTRY)

Created per decap endpoint. This is the **actionable object** that tells the ASIC
to actually decapsulate matching packets. Key attributes:
- `SAI_TUNNEL_TERM_TABLE_ENTRY_ATTR_VR_ID` = virtual router
- `SAI_TUNNEL_TERM_TABLE_ENTRY_ATTR_TYPE` = P2P / P2MP / MP2MP
- `SAI_TUNNEL_TERM_TABLE_ENTRY_ATTR_DST_IP` = local IP (our loopback, packet destination)
- `SAI_TUNNEL_TERM_TABLE_ENTRY_ATTR_SRC_IP` = remote peer (only for P2P/MP2MP)
- `SAI_TUNNEL_TERM_TABLE_ENTRY_ATTR_TUNNEL_TYPE` = `SAI_TUNNEL_TYPE_IPINIP`
- `SAI_TUNNEL_TERM_TABLE_ENTRY_ATTR_ACTION_TUNNEL_ID` = OID of the tunnel object above

For the decap use case, the term entry says:
> "If an outer IP packet arrives with dst=10.0.0.56 and it's IPinIP,
> decapsulate it using the referenced tunnel's TTL/DSCP/ECN modes."

tunnel/term table flow.

## 4. VPP IPIP Tunnel API

Source: `vpp/src/vnet/ipip/ipip.api`

### ipip_tunnel typedef
```c
typedef ipip_tunnel
{
  u32 instance;                              // ~0 = auto
  vl_api_address_t src;                      // outer src IP
  vl_api_address_t dst;                      // outer dst IP (0.0.0.0 for MP)
  vl_api_interface_index_t sw_if_index;      // returned on create
  u32 table_id;                              // underlay VRF ID
  vl_api_tunnel_encap_decap_flags_t flags;   // TTL/DSCP/ECN copy behavior
  vl_api_tunnel_mode_t mode;                 // P2P=0, MP=1
  vl_api_ip_dscp_t dscp;                     // fixed DSCP for encap
};
```

### APIs
- `ipip_add_tunnel` → returns `sw_if_index`
- `ipip_del_tunnel` → takes `sw_if_index`
- `ipip_tunnel_dump` → list all tunnels

### tunnel_encap_decap_flags (from tunnel_types.api)

| Flag | Value | Description |
|------|-------|-------------|
| `ENCAP_COPY_DF` | 0x01 | Copy DF bit from payload to outer |
| `ENCAP_SET_DF` | 0x02 | Set DF bit in outer |
| `ENCAP_COPY_DSCP` | 0x04 | Copy DSCP from inner → outer (uniform mode) |
| `ENCAP_COPY_ECN` | 0x08 | Copy ECN from inner → outer |
| `DECAP_COPY_ECN` | 0x10 | Copy ECN from outer → inner |
| `ENCAP_INNER_HASH` | 0x20 | Hash inner packet for entropy |
| `ENCAP_COPY_HOP_LIMIT` | 0x40 | Copy TTL from inner → outer (uniform mode) |
| `ENCAP_COPY_FLOW_LABEL` | 0x80 | Copy IPv6 flow label |

### tunnel_mode

| Mode | Value |
|------|-------|
| `TUNNEL_API_MODE_P2P` | 0 |
| `TUNNEL_API_MODE_MP` | 1 |

## 5. Mapping SAI → VPP

### SAI Tunnel + Term → VPP IPIP

The SAI tunnel object provides **configuration** (TTL/DSCP/ECN modes).
The SAI term entry provides **match criteria** (dst_ip, src_ip, type).
Together they map to a single VPP IPIP tunnel interface.

```
SAI Tunnel Term Entry                    VPP ipip_add_tunnel
─────────────────────                    ───────────────────
DST_IP = 10.0.0.56        ──────→       src = 10.0.0.56  (we are the tunnel endpoint)
SRC_IP = <remote>         ──────→       dst = <remote>    (P2P only; 0.0.0.0 for P2MP)
TYPE = P2MP               ──────→       mode = TUNNEL_API_MODE_MP

SAI Tunnel Object
─────────────────
DECAP_TTL_MODE = PIPE     ──────→       flags: no ENCAP_COPY_HOP_LIMIT
DECAP_TTL_MODE = UNIFORM  ──────→       flags |= ENCAP_COPY_HOP_LIMIT (0x40)
DECAP_DSCP_MODE = PIPE    ──────→       flags: no ENCAP_COPY_DSCP
DECAP_DSCP_MODE = UNIFORM ──────→       flags |= ENCAP_COPY_DSCP (0x04)
DECAP_ECN_MODE = COPY     ──────→       flags |= DECAP_COPY_ECN (0x10)
```

**Note**: For the decap-only case (which is what `TunnelDecapOrch` creates), the
tunnel is primarily for receiving. VPP still needs an IPIP tunnel interface created
with the local IP as `src` so that it recognizes incoming IPinIP packets and
decapsulates them. The `dst` can be `0.0.0.0` for P2MP (accept from any source).

### VPP CLI Equivalent

For the ASIC_DB state shown above (P2MP, dst_ip=10.0.0.56, pipe TTL/DSCP):
```
create ipip tunnel src 10.0.0.56 dst 0.0.0.0 mode mp
set interface state ipip0 up
```

For a P2P variant (dst_ip=10.0.0.56, src_ip=10.0.0.1):
```
create ipip tunnel src 10.0.0.56 dst 10.0.0.1
set interface state ipip0 up
```

### Step 2: Add IPIP tunnel term handling to TunnelManager

**Files**: `vslib/vpp/TunnelManager.h`, `TunnelManager.cpp`

```cpp
// TunnelManager.h — new data struct
struct IpipTunnelData {
    uint32_t sw_if_index;
    sai_object_id_t tunnel_oid;     // referenced SAI tunnel
};

// TunnelManager.h — new members
std::unordered_map<sai_object_id_t, IpipTunnelData> m_ipip_term_map;

sai_status_t create_ipip_tunnel_term(
    const std::string& serializedObjectId,
    sai_object_id_t switch_id,
    uint32_t attr_count,
    const sai_attribute_t* attr_list);

sai_status_t remove_ipip_tunnel_term(
    const std::string& serializedObjectId);
```

`create_ipip_tunnel_term()` logic:
1. Parse term entry attrs: `TYPE`, `DST_IP`, `SRC_IP` (if P2P), `ACTION_TUNNEL_ID`
2. Look up tunnel object → get `DECAP_TTL_MODE`, `DECAP_DSCP_MODE`, `DECAP_ECN_MODE`
3. Build `vpp_ipip_tunnel_t`:
   - `src` = term entry `DST_IP` (our local IP)
   - `dst` = term entry `SRC_IP` (remote; `0.0.0.0` for P2MP)
   - `mode` = P2P or MP based on term `TYPE`
   - `flags` = mapped from SAI TTL/DSCP/ECN modes
4. Call `vpp_ipip_tunnel_add()` → get `sw_if_index`
5. Set interface up: `interface_set_state(ipip_ifname, true)`
6. (Optional) Bind interface to overlay VRF: `set_interface_vrf()`
7. Store in `m_ipip_term_map`

### Step 3: Hook into SwitchVpp create/remove

**File**: `vslib/vpp/SwitchVpp.cpp`

### Step 4: Relationship with tunterm_acl

The `tunterm_acl` VPP plugin is a **separate mechanism** used for ACL-based decap
(matching `SAI_ACL_ENTRY_ATTR_FIELD_TUNNEL_TERMINATED`). It operates independently
from the IPIP tunnel interface approach.

For the basic IPinIP decap case:
- **VPP IPIP tunnel** handles the actual decapsulation (strip outer IP header)
- **tunterm_acl** may additionally be used for policy-based routing of decapped
  packets (redirect to specific next-hop after decap)

Both can coexist. The IPIP tunnel does the heavy lifting; tunterm_acl adds
fine-grained per-prefix control post-decap.

## 7. Encap IPinIP Tunnel (P2P, MuxOrch / Dual-ToR)

### 7.1 SAI Object Flow

Created by **`MuxOrch`** (`sonic-swss/orchagent/muxorch.cpp`) for dual-ToR MUX cable
scenarios. The tunnel is used to forward traffic to the peer ToR when the MUX is in
standby mode.

Three SAI objects form the encap path:

```
SAI_OBJECT_TYPE_TUNNEL (P2P, with ENCAP_SRC_IP + ENCAP_DST_IP)
         ↑
SAI_OBJECT_TYPE_NEXT_HOP (type=TUNNEL_ENCAP, tunnel_id=above, ip=dst)
         ↑
SAI_OBJECT_TYPE_ROUTE_ENTRY (next_hop_id=above)
```

For the ASIC_DB state `oid:0x2a0000000006a7`:
- `PEER_MODE` = `P2P` (fixed src+dst pair)
- `ENCAP_SRC_IP` = `10.1.0.32` (our loopback)
- `ENCAP_DST_IP` = `10.1.0.33` (peer ToR loopback)
- TTL/DSCP modes both PIPE (don't copy inner → outer)
- `LOOPBACK_PACKET_ACTION` = DROP (prevent encap loops)

### 7.2 Difference from Decap Tunnel

| Aspect | Decap (TunnelDecapOrch) | Encap (MuxOrch) |
|--------|------------------------|-----------------|
| `PEER_MODE` | not set (P2MP implied) | `P2P` |
| `ENCAP_SRC_IP` | optional | **required** |
| `ENCAP_DST_IP` | not set | **required** |
| Tunnel term entry | yes (match incoming) | no |
| Nexthop object | no | yes (`TUNNEL_ENCAP`) |
| Direction | receive & decap | send & encap |
| VPP trigger | tunnel term create | nexthop create |

### 7.3 Current VPP Code Path

When `MuxOrch` creates the `TUNNEL_ENCAP` nexthop, the flow is:

```

  → m_tunnel_mgr.create_tunnel_encap_nexthop()
    → tunnel_encap_nexthop_action()       (TunnelManager.cpp:93)
      → checks SAI_TUNNEL_ATTR_TYPE
      → if VXLAN: create vxlan tunnel + BVI + BD  ✅

```

The existing `tunnel_encap_nexthop_action()` explicitly rejects non-VxLAN:
```cpp

    SWSS_LOG_ERROR("Unsupported tunnel encap type %d in %s", ...);
```

### 7.4 Mapping SAI Encap Tunnel → VPP

The P2P encap tunnel maps directly to a VPP P2P IPIP tunnel:

```
SAI Tunnel Object (P2P encap)            VPP ipip_add_tunnel
─────────────────────────────            ───────────────────
ENCAP_SRC_IP = 10.1.0.32    ──────→     src = 10.1.0.32
ENCAP_DST_IP = 10.1.0.33    ──────→     dst = 10.1.0.33
PEER_MODE = P2P              ──────→     mode = TUNNEL_API_MODE_P2P (0)
ENCAP_TTL_MODE = PIPE        ──────→     flags: no ENCAP_COPY_HOP_LIMIT
ENCAP_DSCP_MODE = PIPE       ──────→     flags: no ENCAP_COPY_DSCP
```

VPP CLI equivalent:
```
create ipip tunnel src 10.1.0.32 dst 10.1.0.33
set interface state ipip0 up
```

Then when routes are programmed pointing at the tunnel nexthop:
```
ip route add <inner_prefix> via <dst_ip> ipip0
```

### 7.5 Implementation Plan for Encap

Extend `tunnel_encap_nexthop_action()` in TunnelManager to handle IPINIP:

```cpp
sai_status_t TunnelManager::tunnel_encap_nexthop_action(
    const SaiObject* tunnel_nh_obj, Action action)
{
    // ... existing code to get tunnel_obj, check type ...

    if (attr.value.s32 == SAI_TUNNEL_TYPE_VXLAN) {
        // existing VxLAN path (unchanged)
    }
    else if (attr.value.s32 == SAI_TUNNEL_TYPE_IPINIP) {
        return ipip_encap_nexthop_action(tunnel_nh_obj, tunnel_obj, action);
    }
    else {
        return SAI_STATUS_NOT_IMPLEMENTED;
    }
}
```

New method `ipip_encap_nexthop_action()`:

1. Get `ENCAP_SRC_IP` and `ENCAP_DST_IP` from the tunnel object
   - For P2P: both are in the tunnel object itself
   - The nexthop's `SAI_NEXT_HOP_ATTR_IP` is the tunnel endpoint IP
     (same as `ENCAP_DST_IP`)
2. Get TTL/DSCP modes → build VPP flags
3. **On CREATE**:
   - Call `vpp_ipip_tunnel_add()` → get `sw_if_index`
   - Set interface up
   - Store `sw_if_index` in `m_tunnel_encap_nexthop_map`
     (same map as VxLAN, keyed by nexthop OID)
4. **On DELETE**:
   - Call `vpp_ipip_tunnel_del(sw_if_index)`
   - Remove from map

The route programming already works — `SwitchVppRoute.cpp` calls
`IpRouteAddRemove()` for tunnel nexthops, which uses `m_tunnel_mgr.get_tunnel_if()`
to resolve the nexthop OID → `sw_if_index`. As long as we store the IPIP tunnel's
`sw_if_index` in `m_tunnel_encap_nexthop_map`, routes will automatically use it.

### 7.6 Key Difference from Decap Path

| | Decap (Section 6) | Encap (Section 7) |
|-|-------------------|-------------------|
| **VPP trigger** | `SAI_OBJECT_TYPE_TUNNEL_TERM_TABLE_ENTRY` create | `SAI_NEXT_HOP_TYPE_TUNNEL_ENCAP` nexthop create |
| **Hook point** | New code in `SwitchVpp::createObject()` | Existing `create_tunnel_encap_nexthop()` path |
| **VPP tunnel src** | term entry `DST_IP` (local) | tunnel `ENCAP_SRC_IP` |
| **VPP tunnel dst** | term entry `SRC_IP` or 0.0.0.0 | tunnel `ENCAP_DST_IP` |
| **VPP mode** | P2P or MP (from term type) | P2P (always, from `PEER_MODE`) |
| **Interface usage** | Receives & decaps | Encaps & sends |

**NOTE**: A single VPP IPIP P2P tunnel can do both encap AND decap simultaneously.
If both the decap term and the encap nexthop point to the same src/dst pair, only one
VPP tunnel interface is needed. The implementation should detect this and reuse the
existing `sw_if_index` rather than creating a duplicate.

## 8. Open Questions

1. **Shared tunnel**: When both decap term (section 6) and encap nexthop (section 7)
   reference the same src/dst IP pair, should we create one VPP tunnel or two?
   VPP's IPIP tunnel is always bidirectional, so one tunnel handles both directions.
   Need a shared lookup map keyed by `(src_ip, dst_ip)` to avoid duplicates.

2. **Overlay VRF binding**: After creating the IPIP tunnel interface in VPP, which
   VRF table should it be bound to? The tunnel object has overlay/underlay RIF OIDs.
   The overlay RIF's VRF should be used for the inner (decapped) packet routing.

3. **IPIP is built-in vs plugin**: Unlike VxLAN (a VPP plugin), IPIP is in
   `vnet/ipip/` (core). The message ID base handling in SaiVppXlate is different:
   - VxLAN: `__plugin_msg_base = vxlan_msg_id_base; VXLAN_MSG_ID(msg)`
   - IPIP: needs its own msg_id_base via `vl_client_get_first_plugin_msg_id("ipip_...")`
     (even core APIs use this pattern — see `vpp_ipip_tunnel.c` example)

4. **P2MP multipoint mode**: In VPP, `mode=MP` with `dst=0.0.0.0` creates a
   multipoint tunnel that accepts IPinIP from any source. This matches the SONiC
   use case where `P2MP` decap terms accept traffic from multiple remote endpoints.

5. **Neighbor entry for encap**: VxLAN encap creates a neighbor entry on the tunnel
   interface (`ip4_nbr_add_del` with `no_fib_entry`) to build the inner Ethernet
   header. For IPIP encap this is **not needed** — IPIP has no inner Ethernet header,
   it's pure IP-in-IP.
