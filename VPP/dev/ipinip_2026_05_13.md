# VPP class-E FIB drop ŌĆö SONiC `sonic_fib_init` plugin

## Problem

`test_decap` on SONiC VPP T0 sends IPIP-encapped packets whose inner
destination falls in `240.0.0.0/4` (e.g. `240.0.0.1`). After successful
decap inside `ipip4-input` the inner packet enters `ip4-lookup`, hits a
hard-coded `dpo-drop`, and is silently dropped ŌĆö even though SONiC has
installed a default route that should forward it.

Root cause: VPP installs a fixed set of high-priority drop entries in
every IPv4 FIB at table creation, in
[`vpp/src/vnet/fib/ip4_fib.c :: ip4_specials[]`](vpp_build2/sonic-buildimage/platform/vpp/vppbld/repo/src/vnet/fib/ip4_fib.c):

| prefix             | source                       | override-able? |
| ------------------ | ---------------------------- | -------------- |
| `0.0.0.0/0`        | `FIB_SOURCE_DEFAULT_ROUTE`   | yes            |
| `0.0.0.0/32`       | `FIB_SOURCE_DEFAULT_ROUTE`   | yes            |
| `224.0.0.0/4`      | `FIB_SOURCE_SPECIAL`         | **no**         |
| `240.0.0.0/4`      | `FIB_SOURCE_SPECIAL`         | **no**         |
| `255.255.255.255/32` | `FIB_SOURCE_DEFAULT_ROUTE` | yes            |

`FIB_SOURCE_SPECIAL` is `FIB_SOURCE_FIRST` in
[`fib_source.h`](vpp_build2/sonic-buildimage/platform/vpp/vppbld/repo/src/vnet/fib/fib_source.h) ŌĆö
the highest-priority FIB source. Neither the `ip_route_add_del` binary
API (`FIB_SOURCE_API`) nor `ip route` CLI (`FIB_SOURCE_CLI`) can win the
longest-prefix match against it. Only an in-process call to
`fib_table_entry_special_remove(fib_index, &pfx, FIB_SOURCE_SPECIAL)`
removes it.

Other SONiC platforms (BCM, MLNX) don't impose this filter, so SONiC's
expectation is "FIB lookup is governed by programmed routes". Fix here:
strip the SPECIAL drop on SONiC VPP.

## Debug method (also added to `aoi/VPP/vpp_trace_packet.md`)

1. Trace the offending packet via `trace add dpdk-input N` + `show trace`.
2. In the `ip4-lookup` frame, look at structural fields, not the textual
   error name: `fib:0 adj:0` and `dpo-idx 0` ŌćÆ this is the global drop DPO.
3. Cross-check with `show ip fib <dst>` ŌĆö the matched entry will read
   `src:special` `[1] [@0]: dpo-drop ip4`.
4. `show errors`: do `clear errors`, reproduce, then `show errors` ŌĆö
   counters increment under whichever ip4 node `b->error` was set in
   (often misleadingly `ip4-local: unknown ip protocol`; the real drop
   site is `ip4-drop` after FIB miss).

## Fix: new plugin `sonic_fib_init`

Pure init-time + small binary API plugin under
`vpp_build2/sonic-buildimage/platform/vpp/vppbld/plugins/sonic_fib_init/`:

| file | role |
| ---- | ---- |
| `sonic_fib_init.c` | `VLIB_MAIN_LOOP_ENTER_FUNCTION` strips `240.0.0.0/4` SPECIAL from fib 0; API handler `vl_api_sonic_fib_strip_specials_t_handler` does the same for any user-supplied `vrf_id` via `fib_table_find(FIB_PROTOCOL_IP4, vrf_id)` |
| `sonic_fib_init.api` | `autoreply define sonic_fib_strip_specials { u32 vrf_id; }` |
| `sonic_fib_init.h`   | `msg_id_base` |
| `CMakeLists.txt`     | `add_vpp_plugin(sonic_fib_init SOURCES sonic_fib_init.c API_FILES sonic_fib_init.api)` |
| `FEATURE.yaml`       | documentation |

Scope intentionally narrow: only `240.0.0.0/4` is removed.
`224.0.0.0/4` (multicast) is left intact to avoid affecting IGMP/PIM.
IPv6 has no equivalent `ff00::/8` SPECIAL drop
(checked [`ip6_fib.c`](vpp_build2/sonic-buildimage/platform/vpp/vppbld/repo/src/vnet/fib/ip6_fib.c)
ŌĆö v6 specials are only the default route and the link-local DPO).

### Multi-VRF coverage

- **fib 0**: handled inside VPP at `VLIB_MAIN_LOOP_ENTER_FUNCTION` (runs
  after all init funcs so the default FIB and its specials exist).
- **VRFs created later** (SAI virtual_router create ŌåÆ SAI shim
  `__ip_vrf_add_del` ŌåÆ `IP_TABLE_ADD_DEL` binary API): the shim invokes
  `sonic_fib_strip_specials { vrf_id }` immediately after a successful
  IPv4 add. `fib_table_entry_special_remove` is **C-only** and not on
  the binary API surface, so the call has to go through this custom
  plugin API.

### Startup wiring

Both startup.conf templates updated to load it:

- [`docker-syncd-vpp/conf/startup.conf.tmpl`](vpp_build2/sonic-buildimage/platform/vpp/docker-syncd-vpp/conf/startup.conf.tmpl)
- [`docker-sonic-vpp/conf/startup.conf.tmpl`](vpp_build2/sonic-buildimage/platform/vpp/docker-sonic-vpp/conf/startup.conf.tmpl)

```
plugin sonic_fib_init_plugin.so { enable }
```

## SAI shim wiring (sonic-sairedis)

`vpp_build2/sonic-buildimage/src/sonic-sairedis/vslib/vpp/vppxlate/SaiVppXlate.c`,
following the existing `tunterm_acl` pattern:

1. Includes for `<vpp_plugins/sonic_fib_init/sonic_fib_init.api{,_enum,_types}.h>`
   plus the four `vl_typedefs`/`vl_endianfun`/`vl_calcsizefun`/`vl_api_version`
   re-include blocks.
2. `static u16 sonic_fib_init_msg_id_base;` and `SONIC_FIB_INIT_MSG_ID(id)`.
3. `vl_api_sonic_fib_strip_specials_reply_t_handler` registered in
   `foreach_vpe_plugin_api_reply_msg`.
4. `get_base_msg_id()` looks up `sonic_fib_init_<ver>`.
5. In `__ip_vrf_add_del()`, on `ret == 0 && is_add && !is_ipv6`:
   `M(SONIC_FIB_STRIP_SPECIALS, smp); smp->vrf_id = htonl(vrf_id); S(smp); WR();`
   inside the existing `VPP_LOCK`. Failure is logged but doesn't fail
   the VRF create (the VRF itself was created successfully).

## How `vl_api_sonic_fib_strip_specials_t` exists

It's not in the tree ŌĆö it's emitted by `vppapigen` at build time from
`.api` files. The pattern `vl_api_<define_name>_t` (and `_reply_t` when
`autoreply`) is generated into
`build-root/.../<plugin>/sonic_fib_init.api.h` and installed to
`/usr/include/vpp_plugins/sonic_fib_init/sonic_fib_init.api.h` by
`vpp-dev_*.deb`. The repeated `#include` blocks in `SaiVppXlate.c` with
different sentinel macros extract: struct definitions (`vl_typedefs`),
endian helpers (`vl_endianfun`), size helpers (`vl_calcsizefun`),
version constant (`vl_api_version`).

## Build chain

Three artifacts span three build stages. Rebuilding only
`docker-syncd-vpp` is **not** sufficient:

| changed source | built by | output |
| -------------- | -------- | ------ |
| `vppbld/plugins/sonic_fib_init/*` | `platform/vpp/vppbld/Makefile` | `vpp-plugin-core_*.deb` (carries `sonic_fib_init_plugin.so`), `vpp-dev_*.deb` (carries generated `.api.h`) |
| `SaiVppXlate.c` | sairedis rules | `libsaivpp` `.deb` ŌĆö must compile against the new `vpp-dev` |
| `startup.conf.tmpl` | docker-syncd-vpp rules | final container |

Two pitfalls:

1. `vppbld/Makefile` downloads pre-built debs from buildkite by default
   (`VPP_DEB_URL`). If the download succeeds, the plugin is **not**
   compiled. Force `build_locally` (or wipe staged `.deb`s under
   `$(DEST)` so the curl fails).
2. The cache mode is `GIT_CONTENT_SHA`. A full top-level
   `make target/docker-syncd-vpp.gz` from `sonic-buildimage` does cascade
   correctly through sairedis and the VPP debs because both source trees
   have changed.

### Post-build verification

```bash
dpkg-deb -c target/debs/bookworm/vpp-plugin-core_*.deb | grep sonic_fib_init
dpkg-deb -c target/debs/bookworm/vpp-dev_*.deb        | grep sonic_fib_init.api.h
```

Then at runtime in the syncd-vpp container:

```
vpp# show plugins | grep sonic_fib_init
vpp# show ip fib 240.0.0.1
# ŌåÆ should match default route, not "src:special ... dpo-drop"

# multi-VRF check
vpp# ip table add 100
vpp# show ip fib table 100 240.0.0.1
# ŌåÆ also no special drop (because SAI shim called sonic_fib_strip_specials)
```

Run `test_decap` ŌĆö Packet 16 should no longer be dropped after decap.

## Open follow-ups

- Decide whether to also strip `224.0.0.0/4` if any future SONiC test
  expects multicast forwarding through the data plane.
- Consider adding a `show sonic-fib-init` debug CLI listing which VRFs
  have been processed.
- If we ever want to drop the plugin entirely, the alternative is a
  one-line upstream patch under `vppbld/patches/` removing
  `240.0.0.0/4` from `ip4_specials[]` ŌĆö covers all VRFs by construction
  with zero runtime cost but diverges from upstream VPP source.
