### Root cause

At boot, orchagent creates the tunnel term entry at 11:40:02, which calls
`create_ipip_vpp_tunnel` → `vpp_sw_interface_find_by_ip(10.0.0.56)`.
But PortChannel101's IP (`10.0.0.56/31`) isn't programmed in VPP until
11:40:04 (via `vpp_add_del_intf_ip_addr_norif` → `interface_ip_address_add_del`).
So `vpp_sw_interface_find_by_ip` fails → `sw_interface_set_unnumbered` is never called
→ tunnel creation fails.

### Solution: deferred unnumbered retry

Instead of failing tunnel creation when the RIF IP isn't ready, we **defer**
the unnumbered operation and retry it when the IP is later programmed.

**Data structure** — `TunnelManager.h`:
```cpp
struct PendingUnnumbered {
    uint32_t sw_if_index;       // tunnel sw_if_index
    vpp_ip_addr_t src_address;  // tunnel source IP
    uint32_t vrf_id;            // VRF for the lookup
};
std::unordered_multimap<std::string, PendingUnnumbered> m_pending_unnumbered;
```
Keyed by source IP string so lookup is O(1) per programmed IP.

**On tunnel creation** — `create_ipip_vpp_tunnel` in `TunnelManager.cpp`:
- If `vpp_sw_interface_find_by_ip` fails, log WARN (not ERROR), enqueue a
  `PendingUnnumbered` entry keyed by the source IP string, and **return SUCCESS**.
  The tunnel is UP in VPP, just missing the unnumbered binding.

**On tunnel removal** — `remove_ipip_vpp_tunnel`:
- Erase any pending entries matching the tunnel's `sw_if_index`.

**Retry hook** — `retry_pending_unnumbered(const vpp_ip_addr_t &rif_ip)`:
- Called from the two active RIF IP programming paths after a successful
  `interface_ip_address_add_del` with `is_add=true`:
  1. `vpp_add_del_intf_ip_addr_norif` — handles Port, PortChannel, Vlan, SubPort
  2. `vpp_interface_ip_address_update` — handles Loopback
- Converts `rif_ip` to string, does `equal_range` on the multimap.
- For each match: `vpp_sw_interface_find_by_ip` → `sw_interface_set_unnumbered`.
  Successful entries are erased; failures remain pending.

**Not hooked**: `vpp_add_del_intf_ip_addr` (dead code — only call site is commented out).
