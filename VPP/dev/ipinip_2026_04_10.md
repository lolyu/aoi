# VPP IPinIP P2MP — 2026-04-10

## The node.c Fix Was Wrong

The original patch (change 45461) inserted a P2MP wildcard fallback in `node.c`:

```c
// After P2P miss:
ip46_address_reset(&key0.dst);
key0.mode = IPIP_MODE_P2MP;
t0 = ipip_tunnel_db_find(&key0);   // hits {mode=P2MP, dst=0} sentinel → always decaps
```

This **breaks the test** `test_mipip4`: after `teib.remove_vpp_config()`, the test calls
`send_and_assert_no_replies` — but with the fix the P2MP sentinel is a wildcard hit, so
the packet decaps instead of dropping → assertion fails.

**The fix is wrong for upstream VPP.** The `{mode=P2MP, dst=0}` DB entry is a
control-plane sentinel for `ipip_tunnel_db_find_by_sw_if_index` bookkeeping, not a
decap catch-all. Unknown peers must drop.

The correct code path is revert to upstream: `P2P → 6RD` (no P2MP step).

## How VPP P2MP Actually Works

### Key construction in `node.c` (per-packet, ipip4-input)

```c
ipip_tunnel_key_t key0 = {
    .mode = IPIP_MODE_P2P,           // always P2P
    .fib_index = ...,
};
ip_set(&key0.dst, &ip40->src_address, true);  // dst = outer packet src (= peer IP)
ip_set(&key0.src, &ip40->dst_address, true);  // src = our local IP

t0 = ipip_tunnel_db_find(&key0);             // lookup {P2P, src=local, dst=peer}
```

Note the naming convention: `key.src/dst` are from the tunnel's perspective (not the
packet's). `key.dst = outer packet source.`

### TEIB registration in `ipip.c`

```c
// ipip_teib_mk_key() comment: "construct the key using mode P2P so it can be found in DP"
ipip_mk_key_i(transport, IPIP_MODE_P2P,
              &t->tunnel_src,       // our local IP
              &nh->fp_addr,         // peer IP  ← from teib entry "peer" field
              fib_index, key);
ipip_tunnel_db_add(t, &key);
```

### Full chain

```
create teib ipip0 peer 10.10.10.1 nh 192.168.1.1
  → ipip_teib_entry_added()
  → key = {P2P, src=192.168.0.1, dst=10.10.10.1}
  → ipip_tunnel_db_add()

packet arrives: outer src=10.10.10.1, dst=192.168.0.1
  → node.c key = {P2P, src=192.168.0.1, dst=10.10.10.1}
  → ipip_tunnel_db_find() → HIT → decap
```

Without TEIB entry: P2P miss → 6RD miss → `no tunnel` drop.

## SAI vs VPP Semantic Mismatch

| | SAI `P2MP` term entry | VPP P2MP tunnel |
|---|---|---|
| Match rule | `dst_ip == local_ip` (any source) | per-peer TEIB key `{P2P, dst=peer_ip}` |
| Unknown peer | decaps freely | drops |
| Security model | permissive | allowlist |
| Per-peer registration | not required | **required** via TEIB |

`saivpp` currently has **no ipip implementation at all** — `TunnelManager.cpp` only
handles `SAI_TUNNEL_TYPE_VXLAN`.

## TEIB Peer vs NH — Data Plane Roles

The two addresses serve **different** roles in the data plane.

| TEIB field | CLI arg | Stored in | Used for |
|---|---|---|---|
| peer | `peer P` | `te_key->tk_peer` | encap adjacency walk (`adj_nbr_walk_nh`) |
| nh   | `nh N`   | `te_nh.fp_addr`   | **decap DB key** (`ipip_teib_mk_key` → `key.dst`) |

`ipip_teib_mk_key` (line 635) builds the tunnel DB key with:
```c
nh = teib_entry_get_nh(ne);                       // te_nh = nh parameter
ipip_mk_key_i(transport, IPIP_MODE_P2P,
              &t->tunnel_src, &nh->fp_addr, ...); // key.dst = nh address
```

`node.c` decap lookup sets `key0.dst = outer_pkt.src`.  For a hit, the outer
packet's source address must equal the TEIB **nh** value, not the peer.

Example from `test_ipip.py`:
```python
teib = VppTeib(self, ipip_if,
               ipip_if._remote_hosts[ii].ip4,  # peer = overlay/FIB dest
               itf._remote_hosts[ii].ip4)       # nh   = outer src of arriving packets

# decap test packet: outer src = nh (itf._remote_hosts[ii].ip4)
IP(src=itf._remote_hosts[ii].ip4, dst=itf.local_ip4) / IP(inner) / ...
```

So:
- **TEIB nh** = the underlay IP that actually sends encap'd packets → `outer.src` at decap
- **TEIB peer** = the overlay IP used for encap adjacency and FIB route target

For SONiC SAI mapping:
```
SAI_NEXT_HOP_TYPE_TUNNEL_ENCAP
  SAI_NEXT_HOP_ATTR_IP → 10.10.10.1   ← becomes TEIB peer (overlay dest / adj walk)
```
The TEIB `nh` = FIB-resolved underlay gateway for that peer.

TEIB entries must be re-programmed when the underlay route to the peer changes.

## Complete P2MP Bring-Up Sequence (CLI, with peers)

```
vpp# create ipip tunnel src 192.168.0.1 p2mp outer-table-id 0
vpp# set interface state ipip0 up
vpp# set interface ip table ipip0 0
vpp# set interface unnumbered ipip0 use bvi1000

# Register each peer (one per remote source IP)
vpp# create teib ipip0 peer 10.10.10.1 nh 192.168.1.1
vpp# create teib ipip0 peer 10.10.10.2 nh 192.168.1.2
```

## Where the Real Fix Belongs (SONiC-VPP)

`saivpp`/`syncd` needs to:
1. Implement `SAI_TUNNEL_TYPE_IPINIP` in `TunnelManager.cpp` — call `create ipip tunnel`
   via VPP API on `SAI_OBJECT_TYPE_TUNNEL` create
2. On `SAI_OBJECT_TYPE_TUNNEL_TERM_TABLE_ENTRY` (P2MP type) — call `set interface state`,
   `set interface ip table`, `set interface unnumbered`
3. On each `SAI_NEXT_HOP_TYPE_TUNNEL_ENCAP` nexthop pointing to ipip0 — call
   `teib_entry_add_del` with `peer = SAI_NEXT_HOP_ATTR_IP` and `nh` resolved from FIB

The `node.c` wildcard fallback (option B) would match SAI permissive semantics but breaks
the upstream VPP test model and should not be submitted upstream.
